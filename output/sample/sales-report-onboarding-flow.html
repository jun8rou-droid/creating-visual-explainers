Onboarding

週次売上レポート自動化
これからどうするか（新卒向け）

図解やチェックリストは見たけれど、自分は次に何をすればいいの？というときの進め方の地図です。順番と役割が分かるように並べています。

現 いまの状態（ざっくり）

  • 設計のたたき台はあります（流れ図・チェックリスト・画面の見本）。
  • Python の小さなツールは、サンプル CSV を読み込んで集計し、AI に渡す用のデータを作るところまで動きます（sales-report-automation/README.md）。
  • まだこれからやることとして、LINE 本送信・定期実行・承認の記録などは、会社の方針と合わせて詰める段階です。

運用の決めごと: JST で 1 週間は月 08:00:00 〜 翌週月 07:59:59。毎週月曜 08:00に CSV を自動送信し、受信でツールが起動。詳細はチェックリストの確定メモ・B 節。

本番で変わること（かんたんに）: いまは手でコマンドを打って試すだけ。本番では 「CSV が届いたら同じ処理を始める」 ようにする。週の名前を毎回打たなくするのはあとから足す仕事。くわしくはリポジトリの sales-report-automation/README.md の見出し「いまの CLI と本番の違い（新卒向け）」。

位置づけ: 「道具の原型」と「やることリスト」はある。本番運用に寄せる作業がこれから、です。

あなたが取るべき流れ（おすすめの順番）

次の 4 フェーズ を上から順に。全部を一人でやる必要はありません。分からないところは上司かエンジニアに相談で大丈夫です。

1

フェーズ 1 — 全体を頭に入れる（半日〜1日）

目的: 「何のために何が動くか」を言葉で説明できるようにする。

順すること
1sales-report-build-steps.html を読む（たとえ話と 8 ステップ）。
2sales-report-flow.html で「一枚の流れ」を眺める。
3用語が気になったら sales-report-glossary.md を拾い読み。

ゴール「CSV → 集計 → AI が文章 → LINE などで送る予定」が自分の言葉で言える。

2

フェーズ 2 — 社内で「決めること」を埋める（数日〜数週間）

目的: プログラムより先にビジネス側のルールを決める。曖昧だと手戻りします。

順すること
1sales-report-automation-checklist.md を開く。
2上司・関係部署と打ち合わせしながらチェックを [x] に。
3CSV の列が違う場合は「どの列が顧客・金額か」をメモし、エンジニアに渡す。

ゴールチェックリストの「決めること」が社内で合意に近づく。

3

フェーズ 3 — 手元で試す（エンジニア主・見学可）

目的: 本番の鍵やデータを使う前に動きと出力を確認する。

順すること
1sales-report-automation の README に従い環境を用意。
2まず --dry-run（API 不要・無料で確認）。
3問題なければテスト用の鍵で文案まで。
4日本語ヘッダの CSV は JSON 設定をエンジニアと確認。

ゴール「この週のファイル → この JSON／文章」が再現できる。

4

フェーズ 4 — 本番に近づける（体制に依存）

目的: 毎週同じ品質で回し、トラブル時に説明できるようにする。

順すること
1誰が・いつまでに CSV を渡すかを決める。
2単価空行の扱い（自動候補か人の承認か）を決める。
3LINE / Slack の送信手順・鍵の置き場をセキュリティ方針に合わせる。
4定期実行（cron / GitHub Actions 等）は会社の許可のもとで。
5監査ログの要否・範囲を決める。

ゴール運用手順がドキュメント化され、引き継げる。

役割のイメージ（新卒の立ち位置）

役割これからやりやすいこと
新卒（非エンジニア寄り）フェーズ 1 の順番を守る。フェーズ 2 でチェックリストを会議資料に。フェーズ 3 はログを読む練習。
新卒（エンジニア）README とコードを追う。フェーズ 3 を担当し、フェーズ 4 の自動実行・API を実装。
上司・現場フェーズ 2 の合意、本番データ、承認フローの最終決定。

  1. 1図で全体を理解する

  4. 4配信・定期実行・記録を本番向けにする
