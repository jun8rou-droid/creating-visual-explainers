# 週次売上レポート自動化 — これからどうするか（新卒向け）

**ブラウザで読む:** [sales-report-onboarding-flow.html](sales-report-onboarding-flow.html)（レイアウト付き）／このファイルは編集用の Markdown 版です。

「図解やチェックリストは見たけれど、**自分は次に何をすればいいの？**」というときの、**進め方の地図**です。専門用語の意味だけでなく、**順番と役割**が分かるように書いています。

**運用の決めごと（同期メモ）:** **日本時間（JST）**。1 週間は**月曜 08:00:00 〜 翌週月曜 07:59:59**。毎週**月曜 08:00**に CSV をツールへ自動送信し、**ツールは CSV 受信をトリガー**に処理。詳細は [sales-report-automation-checklist.md](sales-report-automation-checklist.md) の確定メモ・B 節。

**本番で変わること（やさしい言い方）:** いまはエンジニアが手でコマンドを打って試していますが、本番では **「ファイルが届いたら同じ処理を始める」** ようにします。週の名前（`--week`）を毎回手で入れずに済むようにするのは、あとから足す**別の小さな仕事**です。くわしくはリポジトリの [sales-report-automation/README.md](../../sales-report-automation/README.md) の「いまの CLI と本番の違い」。

---

## いまの状態（ざっくり）

- **設計のたたき台**はあります（流れ図・チェックリスト・画面の見本）。
- **Python の小さなツール**は、サンプル CSV を読み込んで集計し、AI に渡す用のデータを作るところまで動きます（詳細はリポジトリ内 `sales-report-automation/README.md`）。
- **まだこれからやること**として、例えば「LINE に本当に送る」「毎週決まった時間に動かす」「誰がいつ承認したかの記録を残す」などは、会社の方針と合わせて詰める段階です。

つまり、**「道具の原型」と「やることリスト」はある。本番運用に寄せる作業がこれから**、という位置づけです。

---

## あなたが取るべき流れ（おすすめの順番）

次の **4 フェーズ** を上から順に進めると、迷いにくいです。全部を一人でやる必要はありません。**分からないところは上司かエンジニアに相談**で大丈夫です。

### フェーズ 1 — 全体を頭に入れる（半日〜1日）

**目的:** 「何のために何が動くか」を言葉で説明できるようにする。

| 順番 | すること |
|:---:|:---|
| 1 | [sales-report-build-steps.html](sales-report-build-steps.html) を読む（たとえ話と 8 ステップ）。 |
| 2 | [sales-report-flow.html](sales-report-flow.html) で「一枚の流れ」を眺める。 |
| 3 | 用語が気になったら [sales-report-glossary.md](sales-report-glossary.md) の該当だけ拾い読み。 |

**ゴール:** 「CSV が入って、集計されて、AI が文章にして、最後は LINE などで送る予定」が自分の言葉で言える。

---

### フェーズ 2 — 社内で「決めること」を埋める（数日〜数週間）

**目的:** プログラムや AI より先に、**ビジネス側のルール**を決める。ここが曖昧だと後で手戻りします。

| 順番 | すること |
|:---:|:---|
| 1 | [sales-report-automation-checklist.md](sales-report-automation-checklist.md) を開く。 |
| 2 | **上司・関係部署と打ち合わせ**しながら、チェックの項目を `[x]` にしていく（例：週の区切り、必須の列、誰が最終確認するか）。 |
| 3 | 実際の会社の CSV がサンプルと違う場合は、「どの列が顧客で、どれが金額か」をメモし、エンジニアに渡す。 |

**ゴール:** チェックリストの「決めること」が、社内で合意された状態に近づく。

---

### フェーズ 3 — 手元で試す（エンジニア主、新卒は見学・手伝いでも可）

**目的:** 本番の鍵や本番データを使う前に、**動きと出力**を確認する。

| 順番 | すること |
|:---:|:---|
| 1 | `sales-report-automation` の README に従い、仮想環境とライブラリを入れる。 |
| 2 | まず **`--dry-run`** で動かす（AI の API を呼ばず、渡す予定のデータだけ表示。料金もかからない）。 |
| 3 | 問題なければ、テスト用の鍵で実際に文案まで出してみる。 |
| 4 | 日付や列名が日本語の CSV なら、README にある **設定ファイル（JSON）** の話をエンジニアと一緒に確認する。 |

**ゴール:** 「この週のファイルを入れたら、このような JSON／文章になる」が再現できる。

---

### フェーズ 4 — 本番に近づける（会社の体制に依存）

**目的:** 毎週同じ品質で回し、トラブル時に説明できるようにする。

| 順番 | すること |
|:---:|:---|
| 1 | **誰が** CSV を出し、**いつまでに**ツールに渡すかを決める。 |
| 2 | 単価が空の行をどう扱うか（自動候補でよいか、人の承認が必要か）を決め、必要なら画面や手順書にする。 |
| 3 | LINE や Slack に送るなら、公式の送信手順・鍵の管理場所をセキュリティポリシーに合わせて決める。 |
| 4 | 「毎週月曜 9 時」のように **自動で動かす**なら、cron や GitHub Actions など、会社が許した方法で組む（エンジニア作業になりやすい）。 |
| 5 | **監査ログ**（いつ誰が何をしたか）を残すか、どこまで残すかを決める。 |

**ゴール:** 運用手順がドキュメント化され、新卒が入ってきても引き継げる。

## 役割のイメージ（新卒の立ち位置）

| 役割 | これからやりやすいこと |
|:---|:---|
| **新卒（非エンジニア寄り）** | フェーズ 1 の読む順番を守る。フェーズ 2 でチェックリストを会議の資料にし、決まったことをメモする。フェーズ 3 はログを読んで「おかしい」と気づく練習。 |
| **新卒（エンジニア）** | フェーズ 1 のあと README とコードを追う。フェーズ 3 を担当し、フェーズ 4 の自動実行や API 連携を実装する。 |
| **上司・現場** | フェーズ 2 の合意、本番データの取り扱い、承認フローの最終決定。 |

## 一言でまとめると

1. **図で全体を理解する**  
2. **チェックリストで社内ルールを決める**  
3. **手元で dry-run から試す**  
4. **配信・定期実行・記録を本番向けにする**

この順で進めれば、「これからどうしたらいいか」が一本の線になります。途中で設計を変えてもよいので、**分からない項目は空欄のままにせず、誰に聞くかまで書いておく**と、あとからの自分とチームが助かります。
