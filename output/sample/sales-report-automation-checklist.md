## パイプライン概要（設計レビュー反映）

**原則**: 数値・照合・候補抽出は **自動報告ツール内で決定的に**行う。ツール内ストアは **正本ではなくレポート／単価補完用の二次データ**（受注管理ツールのエクスポートは正本のスナップショット）。LLM には **集計済み・候補化済みの構造化データだけ**渡し、要約・通知文・注意書きの**文章化**に寄せる（単価の最終確定は LLM に任せない）。

1. 受注管理ツールから週次 CSV エクスポート（正本のスナップショット）
2. 取り込み: スキーマ検証・正規化・主キー付与・重複検知
3. 永続化: レポート用ストアへ保存（**週キー + `run_id`** などで冪等・再実行可能に）
4. 決定的処理  
   - **4a.** 集計（KPI・内訳）  
   - **4b.** 単価欠損行の候補（履歴ストア参照 → 候補・信頼度・理由コード）
5. 構造化アウトプット（JSON 等）: サマリー・候補・フラグのみ
6. LLM: ナラティブ・配信文案・注意点（**入力された数値を引用**し、再計算しない）
7. 人間レビュー（必要なら単価候補の承認）
8. 配信 + 監査ログ（承認・送信の記録）

### パイプライン確認用チェック

- [x] 正本とツール内蓄積の役割を、社内ドキュメントで **一方向（エクスポート → ツール）** と明記したか（**社内 Notion/Wiki に1ページ作成済**。リポジトリ外のため URL は記載しない。ドラフト元: [sales-report-boundary-and-canonical-draft.md](sales-report-boundary-and-canonical-draft.md)）
- [x] 主キー・重複取込・同じ CSV を再実行したときの **冪等性** を設計したか（ウィザード: **新しい CSV で上書き**、`--replace-week` 相当）
- [x] LLM 入力から **生 CSV 全量・全履歴の丸投げ**を避け、手順 5 の薄いペイロードに限定したか（ウィザード: 集計 JSON 前提／**名称は法務承認のうえでプロンプトに含める**意向）

### 確定メモ · タイムゾーン・週の定義・トリガー（運用予定）

- **タイムゾーン**: **日本時間（JST / Asia/Tokyo）**で扱う。
- **1 週間の定義**: **月曜日 08:00:00** から **翌週の月曜日 07:59:59** までを 1 週間とする（翌週月曜 08:00:00 からは次の週の開始）。
- **CSV の自動送信**: 毎週**月曜 08:00（JST）**に、スケジューラ等から**作成予定のツールへ CSV を 1 回自動送信**する。
- **ツール側のトリガー**: 時刻ではなく **「CSV を受信したこと」**を、取り込み〜下流処理の起点とする（ファイル監視・Webhook・キュー投入などの実装は別途）。
- 受注管理のエクスポートが **月曜 08:00 の送信に間に合う**よう、先行バッチや共有フォルダ運用にするかは、ソースの項で詰める。

## A. 4 パーツ（必須の一次決定）

- [x] **トリガー**: 外部からの CSV 送信 — **毎週月曜 08:00（JST）· 自動**（作成予定のツールへ週 1 回）／**ツール内の起点**は **CSV 受信**

## B. 週次・数値の定義

- [x] 「週」の範囲 — **JST** で **月曜 08:00:00 〜 翌週月曜 07:59:59** を 1 週間とする。CSV に含めるデータ範囲はこのウィンドウに揃える（受注管理のエクスポート条件と一致させる）。祝・境界の細則は必要なら追記する。
- [x] **受信後の処理**: 重複ファイル・再送・部分到着時の挙動（冪等・ロック・通知）→ **ウィザード: 同週の再送は新しい方で上書き**
- [x] 売上の数え方（出荷日ベースで固定か、例外や別基準の有無）→ **ウィザード: 出荷日ベースで固定**
- [x] 前週比・先月比など **比較対象期間** のルールを固定したか → **ウィザード: 前週同条件で固定したい**
- [x] モックの「材質（直近12か月）」など **①②③で期間が違う指標** を利用者向けに説明できるか → **ウィザード追補: 週次でも短く触れ、定義を添える**（メモ: どの材質がよく出ているか確認したい）

## C. データ連携と品質

> **システム境界（用語）:** 「CSV が届く」から「通知メール／ダッシュボードが更新される」までの途中で、**どのプログラムや人の作業がどこまで担当か**の境目です。例: 「`sales_report_automation` が取り込み〜集計〜LLM 用 JSON まで」「別ジョブがメール送信のみ」など、**箱と矢印が1枚で描ける**とチームの認識が揃います。ウィザードでは「意味が分からない」とのメモあり → **図のたたき台を誰が書くか決めたら [x]**。

- [x] 取り込みファイルの **スキーマ検証**（必須列・型・件数の下限/上限）→ **第1段階: 必須列＋日付形式のみ**（ウィザード追補）。件数上下限などは **第2段階で追記**予定
- [x] 重複ファイル・誤ファイルが来たときの挙動（拒否 / 隔離 / アラート）→ **ウィザード追補: 正式ルールは縮小採用（通知＋単価候補系）**（当初メモの「過去比較通知・単価推測提案」を **全部入りではなく** 採用）
- [x] マスタ不整合（得意先コード欠損など）の扱い → **ウィザード: 当該行はスキップ（リスクは運用で周知）**
- [ ] 「取り込み・まとめ」と「週次レポート生成」の **システム境界**（どのコンポーネントが何をするか）（上記注を読んだうえで **図・一覧の担当と期限**を決めたら [x]）

## D. LLM（Anthropic API）の扱い

- [x] **数値はコード側で確定**し、API には集計結果（JSON/表）のみ渡す方針で合意したか（ウィザード: 設計原則に一致）
- [x] プロンプトに含めてよい項目（PII・得意先名・金額明細の粒度）→ **ウィザード: コードに加え名称も可（法務承認済みを前提）** — **実際の承認記録は別途必須**
- [x] マスキングまたは匿名化の要否（法務・個人情報保護方針とのすり合わせ）→ **ウィザード追補: 法務・情シスとの打ち合わせ日程が決まっている／済み**
- [x] ログにプロンプト全文を残さない等の **ログ方針** → **ウィザード: 限定的な環境のみ全文（アクセス管理あり）**
- [x] レート制限・タイムアウト・リトライ回数の目安 → **ウィザード追補: ライブラリ既定＋数回リトライから開始し、あとで調整**
- [x] 失敗時はテンプレ文のみ送る / 送信中止 などの **フォールバック** → **ウィザード: 定型文のみ送る／画面に出す**
