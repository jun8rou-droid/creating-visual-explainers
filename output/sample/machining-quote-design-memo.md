## 入力フロー（対話式）

- **追加・編集とも同じ対話**（E1）
- 案件共通を先に入力 → **＋工程** でタイプ選択 → 必要な項目だけ順に聞く

| 工程 | 対話 | 時間 |
|------|------|------|
| **外径** | 開始φ（任意・空＝材料径）→ 仕上φ → 削り長 | **自動**（T2+D3） |
| **穴** | 穴φ → 深さ（**本数欄なし**） | **自動**（Mα） |
| **溝** | 幅（任意）→ 深さ（任意）→ **秒（必須）** | 秒入力（Y3） |
| **端面** | **秒のみ**（E3） | 秒入力 |
| **その他** | 名称（任意）→ **秒（必須）**（Z1） | 秒入力 |

同じ穴が複数：**行を都度追加** または **複製**（R2 · 2026-05 採用）。

## 計算式

### 外径（T2 + D3）

```
D           ＝ (開始φ ＋ 仕上φ) ÷ 2
n           ＝ 1000 × Vc ÷ (π × D)
削り回数     ＝ ceil((開始φ − 仕上φ) / 2 ÷ ap)
1回あたり時間 ＝ 削り長 ÷ (f × n)
行の時間（分） ＝ 削り回数 × 1回あたり時間
```

- 荒/仕上は **分けない**（材質ごと ap で回数を割る）
- 2行目以降の開始φ：**材料径に戻すことが多い** → 任意欄（S1）

### 穴（Mα）

```
n       ＝ 1000 × Vc穴 ÷ (π × 穴径)
行の時間（分） ＝ 深さ ÷ (f穴 × n)
```

### 手入力（溝・端面・その他）

- 秒 → 内部で分換算
- ペイン3表示：**90秒（1.5分）**（D3）。外径・穴は **分のみ**

### 金額

```
行の金額 ＝ 行の時間（分）÷ 60 × 時間単価（¥/時）
加工費   ＝ 10円単位切り上げ
材料費   ＝ 体積 × 比重 × ¥/kg（B1）、L は入力そのまま（L1）
```

## 品名（PN1 — 2026-05 グリル合意）

**方針:** おおよそは **マスタから選択**。例外的なものだけ **特殊品** / **その他**。

| モード | UI | 保存 |
|--------|-----|------|
| **catalog（通常）** | プルダウンでマスタ品名を選択 | `product_id` → `products.id` |
| **special（特殊品）** | 「特殊品」を選択。補足テキストは **任意** | `product_mode='special'` · `product_label`（空なら表示は「特殊品」のみ） |
| **other（その他）** | 「その他」を選択。**品名テキスト必須** | `product_mode='other'` · `product_label` |

- **マスタ管理** で品名を登録・並べ替え・無効化（材質マスタと同じ別画面）
- 見積画面から **新規品名の即席登録はしない**（マスタ管理へ誘導）
- 社内メモ1行目・顧客コピー文の先頭に **品名表示** を付ける
- 類似案件（Hα+）: **品名一致タグ** · 候補の優先順位に反映（詳細は QL1〜CD1）

### 品名マスタ（`products` 表）

| 列 | 内容 |
|----|------|
| `id` | TEXT PK（内部 ID · 例 `shaft`） |
| `name` | 表示名（例 `シャフト`） |
| `sort_order` | プルダウン並び |
| `active` | 無効化した品名は選択肢に出さない |

### 見積への保存（`quote_revisions` 追記）

| 列 | 内容 |
|----|------|
| `product_mode` | `catalog` / `special` / `other` |
| `product_id` | FK → `products`（catalog 時のみ） |
| `product_label` | 表示用テキスト（special/other 時。catalog 時は `products.name` を denormalize してもよい） |

確定 rev では **`product_mode` + 表示名をスナップショット**（マスタ改名後も過去 rev の表記は変わらない）。

## データ保管（Storage — DB 一本化 · 2026-06 改定）

> **改定:** 2026-06 グリル（NAS+JSON）を **廃止**し、**すべて DB** に置く。  
> 旧構成（`\\NAS\machining-quote\` + JSON）は **採用しない**。  
> 表設計は **DB パターン1**（NAS フォルダ構成の DB 写し）。

### リポジトリ vs DB

| 置き場 | 内容 |
|--------|------|
| **リポジトリ（Git）** | Next.js コード、**マイグレーション SQL**、スキーマ定義（Prisma/Drizzle 等）、設計メモ |
| **DB（本番）** | 見積・マスタ・設定・インデックス・採番の **すべて** |
| **NAS** | **データ置き場としては使わない**（DB バックアップのコピー先として使うのは可） |

| 項目 | 内容 |
|------|------|
| 端末 | 社内 **Win PC** ＋ **iPad** |
| アクセス | ブラウザ → **社内サーバ（Next.js）** |
| データ | **DB が唯一の正本**（端末・NAS JSON に保存しない） |

```

```

### DB エンジン（Storage2 · 2026-06 グリル確定）

**採用: PostgreSQL**

| 質問 | 回答 | 意味 |
|------|------|------|
| 同時利用（いま） | **1人** | 現状は SQLite でも動く |
| DB 管理 | **専任なし** | 初回セットアップ + `pg_dump` 定期実行を手順化 |
| 累計件数 | **500件以上** | どちらも v1 は足りるが、検索・索引は PostgreSQL が安心 |
| 将来の同時利用 | **2〜3人かも（B）** | **決定打** — 同時保存は PostgreSQL 向き |

**SQLite を採用しない理由:** 将来 2〜3 人が同時に下書き保存・版確定すると、SQLite は **書き込み待ち・エラー** が出やすい。

**SQLite の位置づけ:** 開発者 PC の **ローカル試作のみ** は可。本番（社内 Win PC）は PostgreSQL。

| 項目 | 内容 |
|------|------|
| 接続 | 環境変数 **`DATABASE_URL`**（例 `postgresql://…`） |
| 置き場 | **社内 Win PC 上**（同一 LAN）。専用 DB サーバは v1 では不要 |
| バックアップ | Storage14 — `pg_dump` 毎日 |
| ~~`DATA_ROOT`~~ | 不使用 |

### 見積データ（Storage3）

**入力** は常に保存。**金額・時間** は **版を確定した時点** の値をスナップショットとして保存する（過去にいくらで出したかを後から確認するため · Fα 改定）。

| いつ | 入力 | 金額・時間 |
|------|------|------------|
| **下書き**（`rev IS NULL`） | 保存 | 画面は **都度再計算**（保存しない） |
| **版を確定**（`rev = 1,2,…`） | 保存 | **確定時の値を凍結保存**（後から変更しない） |

**確定 rev に保存する（Storage3a）**

| 対象 | 保存内容 |
|------|----------|
| `quote_operations` | 各行 `minutes`, `amount_yen` |
| `quote_revisions` | 段取り按分（1個）、加工費切上げ前後、1個あたり加工/材料/合計、ロット合計 |
| `quote_revisions` | `hourly_rate_at_confirm`（確定時の時間単価） |
| `quote_revisions` | 社内メモ3項目テキスト（N1 を **確定時点で凍結**） |

**確定 rev に保存しない**

| 対象 | 理由 |
|------|------|
| 類似案件リスト | 開いたときに検索・計算 |
| ペイン2 Vc 一時上書き | K2 |

| 列 | 内容 |

### 版管理（Storage4 · Rev2）

| 操作 | DB 動作 |
|------|---------|
| 下書き保存 | `quote_revisions`（`rev IS NULL`）UPSERT + operations。**金額列は NULL のまま** |
| 版を確定 | 計算結果を含め **確定行 INSERT** → 下書き DELETE |
| 再編集 | 最新 **確定 rev のスナップショットを表示**。編集用に新下書きを作るときは **再計算**（最新マスタ） |

- 確定 rev 行は **更新不可**（追記のみ）
- 1案件あたり下書き行は **最大1件**（`rev IS NULL` UNIQUE 相当）

### 案件 ID（Storage4 · I2）

| 段階 | DB |
|------|-----|
| 新規 | `quotes.draft_no = D-0042`、`formal_id = NULL` |
| 初回確定 | `quotes.formal_id = Q-20260607-001`（**UPDATE**。フォルダリネーム相当） |
| 再確定 | `formal_id` 据置、`quote_revisions.rev` のみ増加 |

- 正式 ID 形式: `Q-YYYYMMDD-###`（3桁ゼロ埋め · Storage11）
- `draft_no` は履歴用に **保持**（削除しない）

### 類似案件（Hα — 2026-05 グリル合意）

**目的:** 値付けチェック（今の ¥/個 が過去と比べて妥当か）

**候補条件（M1）:** 材質一致 · **φ完全一致** · **L±5mm**  
**本数:** 一致不要（多い/少ない両方参考）。1行目に必ず表示  
**件数:** 最大 **5件**（1〜4件はあるだけ表示、5件未満で条件を緩めない）

**並び（O4 + M2）:** 一致タグ数が多い順 → 同点なら新しい順。工程数不一致も候補に含めてよい

**1行（R3）:**
```
Q-2025-118 · S45C φ50 L120 · 50本 · ¥1,240/個（今より-¥170）
[材質][φ][L][工程数]   ※本数タグなし
```

**類似度（S2）:** 点数・%なし。**一致したタグのみ**表示

| タグ | 条件 |
|------|------|
| 材質 | 一致（候補条件を満たせば常に付く） |
| φ | 完全一致 |
| L | ±5mm 以内 |
| 工程数 | **完全一致のみ**（E1）。不一致はタグなし |

**クリック（K4）:** 行をインライン展開 → 工程概要 · 段取り · 加工¥/個

**単価比較（P4）:**
- 基本: **¥/個（加工+材料）** の差分 `（今より±¥n）`
- 材料支給案件: **加工¥/個** で比較
- 込み/支給が違う過去案件: **⚠** を付け、比較は参考扱い

**0件（Z1）:** 「類似案件なし（材質·φ·Lが近い過去見積がありません）」— 条件は自動で緩めない

## AI-API — 図面解析 API 連携（2026-05 合意）

**目的:** 図面ドロップ → **社内サーバ** → Vision API → **構造化 JSON** → 入力欄・工程たたき台。**金額はマスタ計算のみ**。

### 原則

| ID | 決定 |
|----|------|
| API1 | **ブラウザから API を直叩きしない**（キーは Next.js のみ） |
| API2 | 返却は **JSON 固定** + フィールドごと `confidence`（high / medium / low） |
| API3 | `low` または `medium` → **黄色要確認**（DWG1 Y2 同型） |
| API4 | 工程提案は **未確定バナー** — 採用 / 却下を `ai_feedback` に記録 |
| API5 | 同一 PDF の再解析は **キャッシュ**（`drawing_file_hash`） |
| API6 | API 障害時 → Tesseract タイトル欄のみ / 手入力フォールバック |
| API7 | **図面ファイルは NAS**、提案 JSON は **DB**（`ai_suggestions`） |
| API8 | **`ANTHROPIC_API_KEY` あり + ファイル添付** → Vision（`vision-analyze.mjs`）· 失敗時デモフォールバック |

### エンドポイント（本番想定）

```
POST /api/drawings/analyze
  Body: { quote_id, drawing_id }  ※ファイルは既に NAS 仮保存済み
  Response: DrawingAnalyzeResponse（下記）
```

**モック v9:** クライアント内 `setTimeout` で API 応答をシミュレート。

## AI-LEARN — 精度向上ロジック（2026-05 合意）

**目的:** 確定 rev を正解として、**提案と正解の diff** から工程・時間の提案精度を上げる。**学習バッチは社内のみ**（教師データを外に出さない）。

### 推論の3層（オンライン）

```
層A  マスタ計算（Vc/f/ap）     … 説明可能な芯 · 不変
層B  類似案件（quote_index）  … データ少の初期主力
層C  学習補正（model_version） … 段取り・時間残差・工程テンプレ
```

**提案時間** = `minutes_master + residual_model(X)`  
**金額** = 既存 `calcTotals()` のみ（AI は単価を直接出さない）。

### フィードバック

| 表 | 役割 |
|----|------|
| `ai_suggestions` | 提案時 JSON · `model_version` · `quote_id` |
| `ai_feedback` | `user_action`: adopt / reject / edit · `diff_json` · `final_revision_id`（版確定後） |

**`diff_json`（自動生成 · `feedback-diff.mjs`）:** 欄の proposed/final · 工程の added/removed/edited · `summary` 集計。モックは採用/却下/確認済み/版確定時に送信。

**`diff_json` 例（旧）:** 工程追加削除 · `minutes` の proposed/final · `setup_minutes` 差分。

### 夜間バッチ

```
1. 確定 rev のみ抽出（rev IS NOT NULL）
2. ai_feedback がある案件は diff を学習重み大に
3. T1 工程構成分類 · T2 段取り回帰 · T3 時間残差
4. hold-out 評価 — 悪化したら model_version を上げない
5. 合格時のみ settings.model_version を更新
```

### 冷スタート

| 確定 rev 数 | 主な推論 |
|-------------|----------|
| 〜30 | マスタ + プリセット |
| 30〜100 | 類似案件 + 頻出テンプレ |
| 100〜 | 残差モデル本番 |
| 図面紐づけ後 | 図面ハッシュを X に追加 |

### やらない

- マスタ（Vc 等）の自動上書き
- 下書きを教師データに混ぜる
- 評価悪化時のモデル自動デプロイ

### 対話学習（AI-DIALOG · 2026-05）

| 層 | 内容 |
|----|------|
| **対話 UI** | ペイン1 右下「AI 対話」— 自然言語で修正指示 |
| **常時記録** | 対話・採用・却下・確認のたびに `localStorage` + `ai_feedback` |
| **次回反映** | `session-learn.mjs` が補正傾向を次の解析に適用（デモ） |
| **本番** | リアルタイムは傾向キャッシュ · 重み更新は夜間バッチ |

**やらない:** キー入力ごとのモデル再学習（不安定 · 高コスト）

**SQL:** `migrations/002_ai_api.sql`
